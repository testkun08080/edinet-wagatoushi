# ビルド工程・データ品質チェック・フレームワークのプラン

サンプルデータ作成時に「不足データを確実にチェックする」「全データを見れるようにする」「あとから修正しやすい構成」を満たすための設計プランです。

> **実装との対応（2026）**  
> - 列定義の単一ソース: `edinet-wrapper/config/screener_columns.json` → ビルド成果物 `edinet-screener/public/data/column_manifest.json`  
> - 品質レポート: `build_screener_data.py` 実行時に `data_quality_report.json` / `.md` を出力  
> - 運用・コマンドの最新手順: [edinet-wrapper-使い方.md](./edinet-wrapper-使い方.md)  
> 以下の YAML 案は当初の設計メモであり、未導入の部分があります。

---

## 1. 目標の整理

| 目標 | 内容 |
|------|------|
| **データ作成時に確実にチェック** | ビルド実行のたびに不足項目を検出し、レポート出力 or 警告で可視化する |
| **サンプルデータで全データを見れる** | どの企業のどの項目が入っているか／欠けているかを一覧で確認できる |
| **修正しやすいフレームワーク** | キー名・列定義を一箇所で管理し、TSV↔フロントの変更が少ない手間で済む |

---

## 2. 現状の課題（簡潔に）

- **キー名が build に直書き** → 列追加・キー変更のたびに `build_screener_data.py` とフロントの `CompanyMetric` 型・`ColumnVisibilityContext` を両方触る必要がある
- **不足の可視化がない** → ビルド後はフロントで「－」を見ないと気づけない
- **TSV の生構造が見えない** → なぜ null になったか（キーがない／期が違う／ネストが違う）を追いにくい

---

## 3. 推奨アーキテクチャ方針

**「TSV をそのままフロントに渡す」は採用しない**ことを推奨します。

- 理由: TSV は期・連結/個別・ネストが複雑で、フロントがそのまま扱うと複雑になり、表示用の正規化がフロントに集中してしまう。また通信量・パースコストが増える。
- 代わり: **ビルドで「正規化済み summaries + company_metrics」を生成する現行フローを維持**しつつ、以下を追加する。
  - **列定義の単一ソース**（YAML/JSON）で「表示項目 ⇔ データソース（TSV のどのセクション・どのキーか）」を一括管理
  - **ビルドの最後に必ず「データ品質レポート」を出力**し、不足を確実にチェック
  - **サンプル作成時は「全データ一覧」レポート**で、TSV 由来の生キーも含めて何が入っているかを見れるようにする

## 4. ビルド工程の全体像（改修後）

```
[data-set]
  TSV + JSON(メタ)  per 企業/期
        │
        ▼
┌───────────────────────────────────────────────────────────────┐
│ Phase 1: パース・正規化（現行 + 拡張）                          │
│  - parse_tsv → summary / pl / bs / cf                         │
│  - _flatten_for_period（CURRENT_KEYS で期を選択）               │
│  - 列定義 config を読み込み、キー名の複数候補に対応（後述）     │
└───────────────────────────────────────────────────────────────┘
        │
        ▼
┌───────────────────────────────────────────────────────────────┐
│ Phase 2: 出力ファイル生成（現行維持）                           │
│  - companies.json                                              │
│  - summaries/{secCode}.json                                    │
│  - company_metrics.json（列定義に従ってマッピング）             │
└───────────────────────────────────────────────────────────────┘
        │
        ▼
┌───────────────────────────────────────────────────────────────┐
│ Phase 3: データ品質チェック（新規） ★毎回必ず実行               │
│  - 全企業 × 全表示項目について null/空 をチェック              │
│  - data_quality_report.json（機械可読）                         │
│  - data_quality_report.md（人間向け一覧表）                     │
│  - オプション: --strict で重要項目の欠損が多い場合に exit 1     │
└───────────────────────────────────────────────────────────────┘
        │
        ▼
┌───────────────────────────────────────────────────────────────┐
│ Phase 4: サンプル用「全データ可視レポート」（新規）             │
│  - sample モード時のみ、または --report 指定時                  │
│  - TSV 由来の「実際に存在したキー」一覧を出力                   │
│  - all_keys_report.json / all_keys_report.md                    │
│  - 必要なら簡易 HTML レポート（オプション）                     │
└───────────────────────────────────────────────────────────────┘
```

## 5. 列定義の単一ソース（config）

**1 ファイルで「フロントの列 ⇔ データの取得元」を定義**し、ビルドもフロントもここを参照する形にします。

### 5.1 配置案

- **edinet-wrapper** 側に置く（ビルドが TSV を読むので）  
  - 例: `edinet-wrapper/config/screener_columns.yaml`
- フロントは **ビルド成果物** として `public/data/column_manifest.json` を生成し、それを読んでもよい（TypeScript の型や列リストの一元化）。  
  または、同じ YAML を edinet-screener にコピー/シンボリックリンクして参照する運用でも可。

### 5.2 列定義のスキーマ案（YAML）

```yaml
# フロントの列ID、ラベル、カテゴリ、データ取得元を一括定義

columns:
  - id: filerName
    label: 会社名
    category: basic
    source: meta
    keys: [filerName]   # company_entry から

  - id: secCode
    label: 銘柄コード
    category: basic
    source: meta
    keys: [secCode]

  - id: equityRatio
    label: 自己資本比率
    category: valuation
    source: summary
    keys: [自己資本比率]
    # 複数候補可
    # keys: [自己資本比率, 自己資本比率（経営指標等）]

  - id: EPS
    label: EPS
    category: valuation
    source: summary
    keys: [１株当たり当期純利益又は当期純損失]

  - id: BPS
    label: BPS
    category: valuation
    source: summary
    keys: [１株当たり純資産額]

  - id: ROE
    label: ROE
    category: valuation
    source: summary
    keys: [自己資本利益率、経営指標等]

  - id: sales
    label: 売上高
    category: performance
    source: summary
    keys: [売上高]

  - id: operatingProfit
    label: 営業利益
    category: performance
    source: pl
    keys: [営業利益]

  - id: 営業CF
    label: 営業CF
    category: cash
    source: summary_or_cf
    keys_summary: [営業活動によるキャッシュ・フロー]
    keys_cf: [営業キャッシュフロー]

  # 計算項目（TSV にキーがない）
  - id: PER
    label: PER
    category: valuation
    source: summary
    keys: [株価収益率]
    computed: true   # 将来的に株価から計算する場合は別扱い

  - id: PBR
    label: PBR
    category: valuation
    source: none
    computed: true

  - id: 時価総額
    label: 時価総額
    category: valuation
    source: none
    computed: true
```

- **source**: `meta` | `summary` | `pl` | `bs` | `cf` | `summary_or_cf` | `none`
- **keys**: 取得に使うキー名のリスト（先頭から順に試し、最初に見つかった値を使う）
- **computed: true**: 現状は null でよい／将来外部API等で計算

ビルド時はこの config を読み、`summary_to_metrics_row` 相当のマッピングを「config 駆動」で行うようにします。  
こうすると、**列の追加・キー名の変更・複数キー対応**が config の編集だけで済み、build のハードコードが減ります。

## 6. Phase 3: データ品質チェック（確実に実行する）

### 6.1 実行タイミング

- **companies.json / summaries / company_metrics を生成した直後、毎回必ず**実行する。
- `--metrics_only` のときも、生成した company_metrics に対して同様のチェックを実行する。

### 6.2 出力物

| ファイル | 用途 |
|----------|------|
| `data_quality_report.json` | 企業（secCode）ごとに、各列が null でないか。列ごとの「入っている企業数」など。CI やスクリプトで利用しやすい。 |
| `data_quality_report.md` | 人間用。企業×主要項目のマトリクス（✓／－）。「不足データまとめ」の更新にも使える。 |

### 6.3 中身のイメージ

- **data_quality_report.json**
  - `by_company`: `{ secCode: { columnId: boolean (has_value) } }`
  - `by_column`: `{ columnId: { filled: number, total: number, missing_sec_codes: string[] } }`
  - `summary`: 全体の欠損率、全 null の列一覧、全 null の企業一覧

- **data_quality_report.md**
  - 表: 行＝企業、列＝主要項目、セル＝✓ or －（または値のサマリ）
  - 末尾に「全社 null の項目」「全項目が少ない企業」をリスト

### 6.4 オプション: --strict

- 重要項目（例: 自己資本比率, EPS, 売上高, 当期純利益 など）について、**欠損率が N% 超**なら `exit(1)` にする。
- サンプルデータを再ビルドしたあと CI で `build_screener_data.py ... --strict` を回し、意図しない欠損が増えたら検知できるようにする。

## 7. Phase 4: サンプルデータの「全データを見れる」ようにする

### 7.1 目的

- サンプル作成時に「TSV から実際に何が取り出せたか」を一覧で確認する。
- キー名のゆれや期の違いで取れていない項目を、ビルド結果と突き合わせやすくする。

### 7.2 出力物（sample モード or --report 時）

| ファイル | 内容 |
|----------|------|
| `all_keys_report.json` | 企業・期ごとに、その期の summary/pl/bs/cf に**実際に存在したキー名のリスト**。フラット化後のキーでよい。 |
| `all_keys_report.md` | 上記を人間が読みやすい形で（企業名・secCode・期・セクション別のキー一覧）。 |
| （任意）`data_overview.html` | 簡易 HTML。企業一覧＋各企業の「入っている列」一覧＋data_quality の要約。ブラウザで開いて確認用。 |

### 7.3 実装のポイント

- パース〜フラット化後の `summary` / `pl` / `bs` / `cf` の `keys()` を集めるだけでよい。
- 列定義 config の「期待するキー」と照らし合わせて「このキーは config のどれに対応するか」「どれがどこの企業で欠けているか」を data_quality と連携させると、不足データの原因が「キーなし」なのか「期の違い」なのかを切り分けしやすい。

---

## 8. フロント側の扱い（修正しやすくする）

- **列の並び・ラベル・カテゴリ**は、可能なら `column_manifest.json`（ビルドが config から生成）を読んで使う。  
  そうすれば、**列の追加・名前変更は config の修正＋ビルドの再実行**だけでフロントに反映される。
- 型定義（`CompanyMetric`）は、`column_manifest.json` から生成するか、または「manifest の id 一覧」と TypeScript の型を一致させる運用にすると、ずれが減る。
- 既存の `ColumnVisibilityContext` は、初期表示列やストレージ連携だけ残し、**列 id・ラベル・カテゴリは manifest 由来**にすると、二重管理がなくなる。

## 10. 実装の優先順位（ロードマップ）

| 順番 | 内容 | 効果 |
|------|------|------|
| 1 | **Phase 3: データ品質チェック**を build_screener_data.py に組み込む（列は現状のハードコードのまま） | データ再作成のたびに不足を確実にチェックできる |
| 2 | **Phase 4: all_keys_report** を sample モードで出力する | サンプルデータで「何が入っているか」を一覧で見れる |
| 3 | **列定義 config（screener_columns.yaml）** を導入し、build のマッピングを config 駆動に変更 | キー名の追加・変更が一箇所で済み、修正しやすい |
| 4 | **column_manifest.json** を生成し、フロントの列リストをここから読むようにする | フロントとデータの整合性が取りやすくなる |
| 5 | （任意）--strict、CI 組み込み、data_overview.html | 品質ゲートと可視化の強化 |

---

## 11. まとめ

- **データ作成時に確実にチェック**: Phase 3 を毎回実行し、`data_quality_report.json` / `.md` で不足を可視化。必要なら `--strict` でビルドを失敗させる。
- **サンプルデータで全データを見れる**: Phase 4 で `all_keys_report` と（任意で）HTML を出力し、TSV 由来のキーと企業×項目の充実度を一覧できるようにする。
- **修正しやすいフレームワーク**: 列定義を **config 1 ファイル**に集約し、ビルドは config 駆動でマッピング。フロントは manifest を参照。キー名変更・列追加は config の編集とビルドの再実行で済むようにする。

このプランに沿って進めれば、サンプルデータ制作時のチェックと、あとからの修正のしやすさを両立できます。
